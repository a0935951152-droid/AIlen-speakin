# SpeakIn — 多對多即時語音翻譯管線：結構及技術棧

> 目標：多對多（N 講者 × L 語言 × M 裝置）即時串流翻譯系統。
> 核心設計：**「大一統」= 一條統一的語意事件匯流排（Segment Event Bus）**，
> 所有輸入端化約成匯流排事件，所有輸出端（VR 字幕、耳機語音、APP 講稿、投影字幕、外部應用）只是訂閱者。
> 成本與聽眾人數脫鉤：計算 ∝ 同時開口人數 S × 訂閱語言數 L；扇出由 SFU/CDN 承擔。

## 1. 事件 Schema（v1.0）

> 這是全系統唯一日後難以更改的合約，所有元件只依賴此 schema，不互相依賴。
> 管線插拔（§6）、技術棧更新（§8）之所以可行，都建立在這個合約之上。

### 1.1 Topic 命名

```
speakin.{session_id}.stt.{speaker_id}          # 原文 segment（STT 輸出）
speakin.{session_id}.text.{lang}               # 翻譯後文字（MT 輸出，含原文 lang）
speakin.{session_id}.tts.{speaker_id}.{lang}   # TTS 中繼資料（音訊本體走 LiveKit 音軌）
speakin.{session_id}.control                   # floor control / presence / 訂閱統計
speakin.{session_id}.x.{stage_id}.{...}        # 客製 stage 的中間 topic（§6 插拔用）
```

### 1.2 SegmentEvent（stt / text 共用）

```jsonc
{
  "schema_ver": "1.0",
  "event_type": "segment.stt",        // segment.stt | segment.mt
  "session_id": "ses_20260611a",      // 一場活動/會議
  "speaker_id": "spk_alice",          // 一人一軌，講者即音軌
  "segment_id": "spk_alice-000123",   // {speaker_id}-{單調遞增序號}
  "rev": 3,                           // partial 修訂號，單調遞增
  "state": "partial",                 // partial | final
  "t_start_ms": 123450,               // 相對 session 起點的音訊時間軸
  "t_end_ms": 125800,
  "wall_ts": "2026-06-11T07:30:00.123Z",
  "src_lang": "zh",                   // 講者原始語言
  "lang": "en",                       // 本事件 text 的語言（stt 事件 = src_lang）
  "text": "Hello everyone, welcome to ...",
  "words": [                          // 可選：詞級時間戳，供字幕逐字上屏/對齊
    { "w": "Hello", "t0": 123450, "t1": 123700 }
  ],
  "conf": 0.92,                       // 信心分數 0~1
  "trace": [                          // 延遲遙測：流經的每個 stage 追加一筆
    { "stage": "stt@1.3.0", "ms": 320 },
    { "stage": "mt@2.1.0",  "ms": 180 }
  ],
  "meta": {}                          // 客製 stage 的擴充欄位（只准加、不准改既有欄位）
}
```

### 1.3 TtsMetaEvent

```jsonc
{
  "schema_ver": "1.0",
  "event_type": "segment.tts",
  "session_id": "ses_20260611a",
  "speaker_id": "spk_alice",
  "segment_id": "spk_alice-000123",   // 與來源 segment 同 ID，跨模態對齊的關鍵
  "lang": "ja",
  "track_id": "lk-track-uuid",        // 對應的 LiveKit 音軌
  "audio_t0_ms": 127100,              // 此段音訊在該音軌上的起點
  "duration_ms": 2350,
  "trace": [ { "stage": "tts@1.0.2", "first_chunk_ms": 210 } ]
}
```

### 1.4 ControlEvent

```jsonc
{
  "schema_ver": "1.0",
  "event_type": "control.floor",      // control.floor | control.presence | control.subs
  "session_id": "ses_20260611a",
  "payload": {
    "floor_speaker": "spk_alice",                 // 當前主發言者（耳機預設跟隨）
    "active_speakers": ["spk_alice", "spk_bob"],
    "lang_subscribers": { "en": 124, "ja": 0 }     // TTS 惰性啟動依據：0 訂閱者不合成
  }
}
```

### 1.5 不變式（Invariants）

1. `state=final` 後該 segment **不可變**；`rev` 只對 partial 遞增。
2. 消費者以 `(speaker_id, segment_id)` 為 key 做 upsert——字幕端收到新 rev 就地覆寫。
3. **TTS 只消費 final**；字幕端消費 partial（允許回改）；講稿端消費 final（可再潤稿重翻）。
4. `segment_id` 貫穿 stt → text → tts，所有輸出可互相對齊回溯。
5. 翻譯每 (segment, 目標語言) 只做一次，與訂閱人數無關。
6. TTS 惰性啟動：`lang_subscribers[lang] == 0` 的 (speaker, lang) 組合不合成。
7. Schema 演進只准**加欄位**（minor bump）；改/刪欄位 = major bump + 雙版本並行過渡期。
8. 客製資料一律放 `meta`，不污染核心欄位。

## 2. 系統結構

```
═══ 輸入平面（成本 ∝ 同時開口人數 S）═══════════════════════
 講者（一人一軌）─ WebRTC ─▶ LiveKit SFU ─▶ VAD 閘控 ─▶ 串流 STT worker
                                          └▶ (共用麥特例: SA-ASR 模組, Phase 6+)
═══ 語意匯流排 ═════════════════════════════════════════════
 NATS JetStream（topic 見 §1.1）
═══ 處理平面（成本 ∝ S × L；stage 可插拔, 見 §6）═══════════
 MT worker  : 訂閱 stt.* → 批次翻譯 → 發布 text.{lang}
 TTS worker : 訂閱 text.{lang} (final only) → 串流合成 →
              以虛擬參與者發布 LiveKit 音軌 + 發布 tts 中繼事件
 [客製 stage]: 任意插入，如術語校正、敏感詞過濾、講稿潤飾…
═══ 分發平面（扇出 ≈ 零計算成本）═══════════════════════════
 VR 眼鏡   ← text.{lang} partial（滾動修正字幕, ~1s）
 耳機      ← LiveKit audio track {speaker×lang}（~2-4s, floor 策略選軌）
 APP 講稿  ← text.*.final（全語言逐字稿）
 投影幕    ← text.en partial
 外部應用  ← WS/Webhook 訂閱事件；推文字進來 = 虛擬講者直入 TTS
```

## 4. 技術棧

| 層 | 選型 | 理由 / 備註 |
|---|---|---|
| 傳輸/SFU | **LiveKit（自建，開源）** | WebRTC 房間模型 = 音軌交換機；千人扇出；TTS 以虛擬參與者發軌 |
| 事件匯流排 | **NATS JetStream** | 輕量、topic 萬用字元訂閱、持久化回放；單 binary 好部署 |
| STT | **WhisperLiveKit**（SimulStreaming/AlignAtt，多語） | faster-whisper large-v3-turbo INT8 底座；英文低延遲備選 Nemotron Speech Streaming 0.6B |
| 翻譯 | **vLLM + Qwen 系 7B 級 instruct** | 批次吞吐高；prompt 控制術語一致性 |
| TTS | **CosyVoice 2 (0.5B)** | 串流首音 ~150ms、中英日韓；輕量備選 Kokoro-82M |
| 後端框架 | Python 3.12 + FastAPI + Pydantic v2 | schema 即程式碼；async 原生 |
| 前端 | TypeScript + Vite + React + livekit-client | SDK 型別由 JSON Schema 自動生成 |
| 部署 | Docker Compose（dev）→ K8s + kustomize（staging/prod） | 各平面獨立 autoscale |
| 觀測 | 事件 `trace` + Prometheus / Grafana / Loki | 首字延遲、首音延遲、stage 級分解 |
| CI/CD | GitHub Actions + 容器 registry | 回放迴歸測試做品質閘門（§8.2） |
| GPU serving 擴展期 | VoxServe 式 streaming-aware 批次排程 | arXiv:2602.00269 |

## 6. 管線插拔架構（插入 / 更改 / 替換）

> 原理：因為 stage 之間**只透過匯流排 topic 溝通、只依賴事件 schema**，
> 所以「管線」只是一份 YAML 拓撲，改管線 = 改 YAML，不動程式碼。

### 6.1 Stage 合約

每個 stage 是一個目錄，含 `manifest.yaml` + 實作，繼承 `core.stage.Stage`：

```yaml
# stages/mt_vllm/manifest.yaml
id: mt_vllm
version: 2.1.0
consumes: { event: segment.stt, states: [partial, final] }
emits:    { event: segment.mt }
resources: { gpu_mb: 15000, replicas_per: "active_speakers" }
config_schema:                 # 此 stage 可調參數（runner 會驗證）
  model: { type: str, default: "Qwen2.5-7B-Instruct" }
  target_langs: { type: list }
```

```python
class Stage(ABC):
    async def setup(self, cfg): ...
    async def on_event(self, ev: SegmentEvent) -> AsyncIterator[SegmentEvent]: ...
    async def teardown(self): ...
    # core 負責：訂閱/發布、rev 排序、trace 追加、健康檢查、優雅關閉
```

### 6.2 管線 = YAML 拓撲

```yaml
# pipelines/with-glossary.yaml
pipeline: with-glossary
nodes:
  - { id: stt,      use: stt_whisperlive@1.3 }
  - { id: glossary, use: plugins/glossary_corrector@0.2,   # ← 插入
      after: stt }                                          # 自動改寫 topic 接線
  - { id: mt,       use: mt_vllm@2.1, after: glossary,
      config: { target_langs: [en, ja] } }
  - { id: tts,      use: tts_cosyvoice2@1.0, after: mt }
```

三種操作都只動 YAML：
- **插入**：加一個 node + `after:`，runner 自動把中間流量改走 `x.{stage_id}` topic
- **更改**：改 node 的 `config:`（runner 滾動重啟該 stage，不斷流）
- **替換**：`use: tts_cosyvoice2@1.0` → `use: tts_kokoro@0.9`（介面相同即可互換）

### 6.3 多語言路由（route_by）——語言數是設定值，不是結構

擴到 26 語時唯一的實質工程點在 TTS（沒有單一開源模型全語覆蓋），
解法不是改框架，而是節點級路由表——runner 按事件欄位把流量分到不同實作：

```yaml
  - id: tts
    route_by: lang
    routes:
      "[zh, ja, ko, yue]": tts_cosyvoice2@1.0
      "[en, fr, es, de, it, pt]": tts_kokoro@0.9
      "*": tts_mms@0.1            # 低資源語言兜底
    after: mt
```

- MT 同理：低資源語言可路由到更大模型或外部 MT API
- 算力 ∝ S×L 線性成長；大語言數時惰性啟動（§1.5-6）從優化變成必需品
- runner 自 Phase 1 起保留 route_by 語法（完整實作於 Phase 3 TTS 接入時）

### 6.4 影子比對（替換前的安全網）

```yaml
  - id: mt_shadow
    use: mt_vllm@3.0-rc1
    taps: stt                  # 旁路訂閱，不影響主線
    emits_to: shadow.mt        # 輸出進影子 topic，離線比對 BLEU/延遲
```

| 資料 | tests/replay 黃金音檔 | 回放迴歸 + 影子流量 | 真實流量 |
| 前端 | Vite dev server + 熱重載，連本地 gateway | 建置產物，連 staging gateway | CDN 靜態託管 + 版本化 API |
| 設定 | `.env.dev` / compose override | kustomize overlay `staging/` | kustomize overlay `prod/`，secrets 走 K8s Secret/SOPS |
| 進入條件 | — | CI 綠燈（unit + replay 閘門） | staging 跑滿迴歸 + 延遲 SLO 達標 + 手動核准 |
| 觀測 | console + 本地 Grafana | 完整 Prom/Grafana/Loki | 同 staging + 告警（SLO：首字 1.5s / 首音 4s P50） |

要點：
- **同一份容器映像走完三環境**，差異只在 kustomize overlay 與環境變數，杜絕「dev 能跑 prod 不能」
- LiveKit/NATS 在 dev 是 compose 服務，在 staging/prod 是獨立 StatefulSet；介面完全一致
- 前端 SDK 的 API base / LiveKit URL 全部由環境注入，程式碼零分支
- session 是天然的隔離單位：staging 可以掛在 prod 同基建上用獨立 session 跑影子驗證（成本省一套 GPU）

### 7.4 CI/CD 流程

```
push → lint/typecheck/unit → build images
     → dev 自動部署 → replay 迴歸（WER/BLEU/延遲與基線比對，超閾值即 fail）
     → staging 自動部署 → 影子流量 24h + 負載測試
     → 手動核准 → prod canary（單 stage 灰度, §8.2）→ 全量
```

## 8. 技術棧更新機制

> 模型一年換好幾代是常態（本文件選型本身就會過時），所以「可更新」是架構需求不是運維雜務。

### 8.1 模型登錄庫（Model Registry）

```
/nas-data/allen/models/                            # 登錄庫：審核過才晉升進來，版本不可變
├── stt/whisper-large-v3-turbo-int8/{1.0, 1.1}/   # 權重 + sha256 指紋 + 評測報告
├── mt/qwen2.5-7b-instruct-awq/...
└── tts/cosyvoice2-0.5b/...

/nas-data/allen/cache/huggingface/                 # HF 下載暫存（可隨時清），與登錄庫分離
```
- stage manifest 用 `model@版本` 釘選；換模型 = 改 manifest 一行，CI 自動跑回放閘門
- 權重不進 git；以 sha256 指紋鎖定，dev/staging/prod 拉同一份

### 8.2 升級路徑（任何 stage / 模型 / 框架皆同一流程）

1. 新版本以**影子 stage**（§6.4）掛上 staging 真實流量，自動比對品質與延遲
2. 通過 → prod **canary**：新版 stage 只服務 5% session（runner 按 session_id 分流）
3. 指標平穩 → 全量；異常 → runner 一鍵切回舊版（舊版容器保留待命）
4. 回放黃金集隨升級擴充：每次線上事故的音檔都進 `tests/replay/golden/`

### 8.3 依賴與合約的更新紀律

- 事件 schema：只加不改（§1.5），major 變更需雙版本並行一個迭代
- Python 依賴：uv lockfile，每月開 PR 升級 + replay 閘門擋回歸
- 替換整個技術選型（如 NATS → Redpanda、LiveKit → 其他 SFU）：core/bus 與 LiveKit 操作都封裝在 core/ 抽象層後面，stages 與 frontend 不直接 import 廠商 SDK
